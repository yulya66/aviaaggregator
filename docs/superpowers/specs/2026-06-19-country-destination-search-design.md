# Поиск «из города → в любой город страны»

Дата: 2026-06-19
Статус: согласован, готов к планированию

## Цель

Дать пользователю выбрать в поле «Куда» не только конкретный город, но и **страну**, и
получить самые дешёвые рейсы из его города в каждый город этой страны. Расширяем
существующую вкладку «Я знаю куда хочу», а не добавляем новый режим — для пользователя это
одна ментальная модель «куда», просто разной точности (Рим vs Италия) в одном поле.

## Ключевой факт (проверено)

Data API Travelpayouts в параметре `destination` принимает **либо IATA-код города (3 символа),
либо код страны (2 символа)** (Aviasales Data API, Travelpayouts Data API docs). Значит:

- «из города в любой город страны» делается **одним** запросом `prices/latest` с
  `destination = <2-буквенный код страны>` — список городов страны разворачивать не нужно.
- Страну от города отличаем по длине кода: **2 символа = страна, 3 = город**. Отдельный флаг
  в форме не требуется.

## Архитектура

### 1. Данные о странах — `src/data/countries.ts` (новый)

Карта `код страны (2 буквы) → русское имя`, по аналогии с `src/data/cities.ts`,
сгенерированная из TP `countries.json (ru)`
(`https://api.travelpayouts.com/data/ru/countries.json`). Используется для подсказок в
автокомплите и для подписей в выдаче («в любой город **Италии**»).

### 2. Автокомплит «Куда» знает страны

`/api/cities` (GET) ищет и по городам, и по странам:

- При совпадении страны показываются **выше** городов (печатаешь «ита» → первой строкой
  «Италия», ниже — города).
- Страна помечается визуально — бейдж «страна» вместо моноширинного IATA-кода у города.
- В форму, как и сейчас, уходит только код (`<input type="hidden">`). Для страны он
  2-буквенный — этого достаточно, чтобы серверная логика её распознала.

Ответ `/api/cities` для каждого элемента включает тип (`city` | `country`), чтобы клиент
отрисовал бейдж и порядок.

### 3. Логика поиска — `src/app/page.tsx`, режим `route`

`dest` распознаём по длине: 2 символа → страна.

Когда `dest` — страна:

- **город + страна** → один запрос `pricesLatest({ origin, destination: <country> })`.
- **(пусто) + страна** → разворачиваем по нашим домашним хабам (`HOME_HUBS`: SVX, MOW, LED,
  CEK, PEE, TJM, KUF): по запросу `pricesLatest({ origin: hub, destination: <country> })` на
  хаб, **топ-50 самых дешёвых на хаб** (`limit: 50`), результаты сливаем. Это вариант
  «откуда угодно», но origin ограничен нашими городами, а не всей планетой — иначе выдача
  забивается нерелевантными вылетами из чужих стран.

Дальше — как в текущем «одностороннем» поиске:

- группировка: один (самый дешёвый) рейс на маршрут, ключ `${origin}_${destination}`;
- сортировка по цене, фильтр по датам «с/по» (`from`/`to`);
- ползунок цены масштабируется под реальные цены выдачи (`routeCap`);
- каждая карточка — конкретная пара город→город с датой, поэтому партнёрские deeplink'и
  (`buildAviasalesLink`) формируются **как сейчас**, без изменений.

**Упрощение (осознанное):** для страны кэш сделок из БД **не подмешиваем** — в таблицах
`deals`/`anomalies` направления хранятся кодами городов, а не стран, и сопоставить их без
карты «страна→города» нельзя. Страна = только живые цены TP. Сценарии «город+город» и
«город» работают по-прежнему, с кэшем БД.

### 4. Тексты

Заголовки и пустые состояния учитывают страну, например:
«Из города Екб в любой город Италии — пока нет находок. Попробуйте другие даты.»

## Что намеренно НЕ делаем (YAGNI)

- Не строим карту «страна → список городов».
- Не подмешиваем кэш БД для страновых запросов.
- Не трогаем вкладку «Я хоть куда» и cron-джобы (L1/L2/L3).
- Не добавляем отдельную вкладку/режим.

## Сценарии (итог)

| Откуда | Куда | Поведение |
|--------|------|-----------|
| город | город | как сейчас — календарь дат по точному маршруту (+ кэш БД) |
| город | **страна** | из этого города в любой город страны (1 запрос TP) |
| (пусто) | **страна** | из наших 7 хабов в любой город страны (7 запросов, топ-50/хаб) |
| город | (пусто) | как сейчас — все рейсы из города |
| (пусто) | город | как сейчас — все рейсы в город откуда угодно |

## Риски

- TP может вернуть мало/шумные данные по некоторым странам — отсекается ползунком цены и
  фильтром дат; пустое состояние информативно.
- 7 параллельных запросов при «пусто + страна» — приемлемо (`Promise.allSettled`, как уже
  сделано для календаря), частичный сбой хаба не валит выдачу.
