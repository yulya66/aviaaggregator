# Поиск «туда-обратно» в маршрутной вкладке

Дата: 2026-07-03
Статус: черновик, на согласование

## Цель

Дать пользователю искать перелёт **туда-обратно** — чтобы сервис был удобным для обычных
поездок (большинство летает с возвратом). Добавляем режим в существующую вкладку «Я знаю куда
хочу», не создавая новый экран.

## Скоуп (осознанно узкий)

- Только вкладка **«Я знаю куда хочу»** (режим `route`).
- Цены round-trip берём **живым запросом к Travelpayouts** — их нельзя кэшировать как one-way
  (комбинаций «дата туда × дата обратно» слишком много).
- **НЕ трогаем:** ленту «Я хоть куда», таблицы `deals`/`anomalies`, cron-джобы L1/L2/L3,
  односторонний поиск. Всё это остаётся one-way ровно как сейчас.

## Ключевой факт

TP-клиент уже отдаёт обратную дату: `pricesCheap` (`/v1/prices/cheap`) возвращает поле
`return_at`, а `TpLatestPrice.return_date` уже есть в типе. Значит round-trip не требует новой
модели данных — только нового вызова и проброса `return_date` до карточки и ссылки.

## Архитектура

### 1. Переключатель режима — форма `route`

Двухкнопочный тумблер, как у агрегаторов: **«В одну сторону»** (по умолчанию) / **«Туда-обратно»**.
В форму уходит скрытое поле `trip=round` (или его нет = one-way). Сервер читает `sp.trip`.

### 2. Календарь — переиспключаем `RangeCalendar`

Компонент уже есть и не меняется по сути, меняются только подпись и смысл двух дат:

- **one-way:** «Даты вылета» — окно `from`/`to` (как сейчас).
- **round-trip:** «Туда — обратно» — `from` = дата туда, `to` = дата обратно.

Календарю добавляем проп `mode`/`label`; наружу он по-прежнему отдаёт `from`/`to`.

### 3. TP-клиент — новый вызов round-trip

Новая функция `pricesRoundTrip({ origin, destination, departDate, returnDate })`:

- endpoint с `one_way=false` и обеими датами (`departure_at` + `return_at`); точный endpoint
  (`/v3/prices_for_dates` или `/v1/prices/cheap` с фильтром по `return_at`) подтвердим при
  реализации — оба уже возвращают `return_at`.
- Возвращает `TpLatestPrice[]` с заполненным `return_date`. Берём **самые дешёвые** варианты.

### 4. Логика `page.tsx`, режим `route`

Когда `trip=round` и заданы origin + dest + туда (`from`) + обратно (`to`):

- вызываем `pricesRoundTrip`, группируем/сортируем по цене (как сейчас one-way);
- **требуем конкретный город назначения** (город→город). Round-trip «в любой город страны»
  в v1 не делаем — бесконечная комбинаторика дат; отметим как будущее.
- кэш БД не подмешиваем (как для страны) — round-trip только живой TP.

Односторонний режим (`trip` не задан) работает ровно как сегодня — ничего не меняется.

### 5. Карточка и партнёрская ссылка

- `FeedCard`/`DealCard` получают опциональные `returnDate` / `returnLabel`; в round-trip
  карточка показывает **две даты** («12.08 ↔ 20.08») и цену за перелёт туда-обратно.
- `buildAviasalesLink` расширяем параметром `returnDate` — при round-trip deeplink «Купить»
  открывает поиск туда-обратно на Aviasales (даты туда+обратно в URL).

### 6. Тексты

Заголовки/пустые состояния учитывают режим: «Туда-обратно {Екб} → {Стамбул}, 12–20 авг —
пока нет находок».

## Что намеренно НЕ делаем (YAGNI)

- Не делаем round-trip в ленте и в поиске «по стране».
- Не трогаем cron/БД/односторонний поиск.
- Не кэшируем round-trip цены.
- Не добавляем гибкий возврат («обратно ±3 дня») — только точные две даты в v1.

## Сценарии (итог)

| Режим | Откуда | Куда | Даты | Поведение |
|-------|--------|------|------|-----------|
| one-way | (как сейчас) | | окно `from`/`to` | без изменений |
| **round-trip** | город | город | туда + обратно | живой TP round-trip, карточка с 2 датами |
| round-trip | город | страна | — | **не поддерживаем в v1** (подсказка выбрать город) |

## Риски

- Точный TP-endpoint для «точные туда+обратно» надо подтвердить при реализации (fallback —
  `/v1/prices/cheap` + фильтр по `return_at`).
- Данных по некоторым парам/датам может не быть — отсекается пустым состоянием.
- Deeplink round-trip: проверить формат ссылки Aviasales с обратной датой.
